# V3 开发计划

本文档用于明确 `multi_agent_debate` 第三阶段开发范围、实现顺序和验收标准。V3 基于 V2 的文章库、快速话题辩论、后台执行和过程可视化继续开发，目标是把项目从“功能可用”推进到“用户更容易理解、结果更容易带走、失败更容易恢复、分析质量更高”的产品化版本。

## 一、版本目标

V3 的核心目标：

- 增加产品介绍首页和新手路径，让首次访问用户能快速理解产品定位、两类辩论模式和基本使用流程。
- 增加 Markdown 报告导出能力，让辩论结果可以被复制、归档和二次编辑。
- 增加失败恢复和整场重新运行能力，降低真实 LLM 调用失败带来的中断成本。
- 升级文章库体验，支持更清晰地管理普通文章和话题记录。
- 增强辩论详情页可视化，让用户更直观地理解阶段进度、角色交锋、结论和可信度。
- 增强 Agent 输出结构，为前端摘要、争议点视图和报告导出提供稳定字段。
- 增加轻量配置能力，让用户可以选择辩论深度和输出风格。

V3 不包含：

- 用户登录、权限、多用户隔离。
- 生产级任务队列，例如 Celery、Redis 队列。
- 联网搜索、PDF/OCR、多模态输入。
- PDF 导出；V3 只实现 Markdown 导出。
- 报告中的继续追问环节。
- 从某个失败阶段继续运行；V3 只实现整场重新运行。
- 更换默认模型供应商。

## 二、技术栈与运行约定

V3 继续沿用 V2 技术栈：

- 后端：FastAPI、SQLModel、SQLite、Pydantic v2、pydantic-settings、OpenAI-compatible SDK、pytest。
- 前端：Next.js 15 App Router、React 19、TypeScript、Tailwind CSS、ESLint。
- 数据库：默认本地 SQLite，文件名 `debate_assistant.db`。

涉及 LLM API、模型接入或 Agent 底层模型配置时，继续默认使用 DeepSeek OpenAI-compatible 方案：

- API key：只从环境变量 `DEEPSEEK_API_KEY` 读取。
- Base URL：`https://api.deepseek.com`。
- 默认模型：`deepseek-v4-pro`。
- Python SDK：`openai.AsyncOpenAI`。

如果后续使用 LangChain，默认使用 `langchain_openai.ChatOpenAI` 接入。不要把真实 key 写入项目文件、测试、提交记录或文档。

## 三、产品介绍首页和新手路径

V3 首页应从单纯的最近辩论列表升级为“产品介绍 + 工作台入口”。

建议保留路径：

```text
/
```

首页展示内容：

- 产品一句话定位：本地文章审读与多 Agent 观点辩论助手。
- 两类核心模式：
  - 文章深度辩论：围绕文章主张执行 9 阶段分析。
  - 快速话题辩论：围绕一个话题执行 5 阶段轻量讨论。
- 简洁流程说明：输入内容 -> Agent 分析 -> 阶段辩论 -> 裁判报告 -> 导出 Markdown。
- 清晰主入口：
  - 新建文章辩论。
  - 新建话题辩论。
  - 进入文章库。
  - 查看最近辩论。
- 示例入口：
  - 提供 1 到 3 个示例话题，用户点击后可预填快速话题辩论表单。

实现建议：

- 首页第一屏直接说明产品价值，避免用户进入后只看到历史列表。
- 最近辩论仍保留，但放在工作台区域。
- 不做营销型长页面，重点服务本地工具的理解和启动效率。

## 四、Markdown 报告导出

V3 只实现 Markdown 导出，不实现 PDF 导出。

报告应面向“可归档、可复制、可二次编辑”的使用场景。

### 4.1 导出入口

建议在辩论详情页增加：

- `导出 Markdown` 按钮。
- `复制 Markdown` 按钮。

导出内容根据当前辩论详情生成，不需要额外调用 LLM。

### 4.2 报告内容结构

Markdown 报告建议包含：

- 标题。
- 辩论类型：文章辩论或话题辩论。
- 文章来源或话题背景。
- 用户关注问题。
- 辩论状态。
- 裁判结论。
- 可信度评分。
- 正方最强论点。
- 反方最强论点。
- 关键分歧。
- 阶段发言记录。
- 生成时间。

报告不包含：

- 继续追问环节。
- 额外生成的新内容。
- PDF 专用样式。

### 4.3 实现策略

优先在前端根据 `DebateRead` 数据生成 Markdown 字符串：

- 实现 `frontend/lib/report.ts`。
- 输入完整辩论详情对象。
- 输出 Markdown 文本。
- 下载时使用浏览器 Blob。
- 复制时使用 Clipboard API。

如果后续需要后端导出接口，再增加 `GET /api/debates/{debate_id}/report.md`，V3 初版可先不做。

## 五、失败恢复和整场重新运行

V3 增加整场重新运行能力，用于处理真实 LLM 调用失败、输出格式异常、多轮调用超时等情况。

### 5.1 用户行为

在辩论详情页，当辩论状态为 `failed` 时显示：

- 失败原因。
- 已完成阶段数量。
- `重新运行整场辩论` 按钮。

可选增强：已完成但用户不满意结果时，也允许重新运行整场辩论。

### 5.2 后端策略

推荐新增接口：

```text
POST /api/debates/{debate_id}/rerun
```

行为：

- 校验辩论存在。
- 删除该辩论已有 `agent_messages`。
- 清空旧的 `final_report`、`winner`、`credibility_score`、`error_message`。
- 将状态设置为 `running` 或 `pending`。
- 复用 V2 后台任务执行整场辩论。
- 根据原文章的 `source` 判断执行文章 9 阶段或话题 5 阶段。
- 立即返回更新后的 `DebateRead`。

V3 不实现从失败阶段继续运行，避免上下文拼接、阶段幂等和中间结果一致性问题过早复杂化。

### 5.3 前端策略

前端新增：

- API client：`rerunDebate(debateId)`。
- 详情页按钮和确认提示。
- 重新运行后继续复用轮询逻辑。
- 重新运行时显示清晰状态，避免用户误以为旧结果仍有效。

## 六、文章库体验升级

V3 在 V2 文章库基础上增强检索、筛选和信息密度。

### 6.1 列表筛选

文章库新增筛选：

- 全部。
- 普通文章。
- 话题。

筛选逻辑可先在前端完成；如果文章数量增加，再增加后端查询参数。

建议后续接口形态：

```text
GET /api/articles?kind=article
GET /api/articles?kind=topic
```

### 6.2 搜索与排序

文章库新增：

- 标题搜索。
- 来源搜索。
- 关注问题搜索。
- 按创建时间排序。
- 按最近辩论时间排序。
- 按最近可信度评分排序。

V3 初版可先实现前端本地搜索和排序，保持后端简单。

### 6.3 卡片信息升级

文章列表项建议展示：

- 类型标记：文章或话题。
- 标题。
- 来源或话题背景摘要。
- 关注问题。
- 关联辩论数量。
- 最近一次辩论状态。
- 最近一次裁判结论。
- 最近一次可信度评分。
- 操作：查看详情、开始新辩论、删除。

## 七、辩论详情页可视化增强

V3 详情页重点从“能看到消息”升级到“能理解交锋结构”。

### 7.1 顶部摘要区

详情页顶部新增摘要区：

- 辩论类型。
- 当前状态。
- 阶段完成度。
- 裁判结论。
- 可信度评分。
- 正方最强论点摘要。
- 反方最强论点摘要。

运行中辩论展示当前等待阶段；失败辩论展示失败原因和重新运行入口。

### 7.2 阶段可视化

在 V2 阶段进度基础上增强：

- 文章辩论展示 9 阶段时间线。
- 话题辩论展示 5 阶段时间线。
- 每个阶段展示角色、阶段名称、完成状态。
- 当前阶段高亮。
- 失败阶段显示错误状态。

### 7.3 角色交锋视图

新增一个轻量“交锋概览”模块：

- 正方关键论点。
- 反方关键论点。
- 主持人总结的关键分歧。
- 裁判判定依据。

该模块依赖 V3 Agent 输出结构增强。如果旧辩论没有这些字段，前端应优雅降级为隐藏或显示“暂无结构化摘要”。

### 7.4 阅读模式

详情页可增加视图切换：

- 总览。
- 完整过程。
- 裁判报告。

这三个视图不需要拆成独立路由，可在详情页内用本地状态切换。

## 八、Agent 输出质量增强

V3 需要调整 Agent 输出 schema 和 prompt，使每个阶段除了正文文本，还能提供可视化和报告导出需要的结构化字段。

### 8.1 通用输出字段

建议所有阶段输出包含：

```json
{
  "summary": "本阶段一句话摘要",
  "key_points": ["关键点 1", "关键点 2"],
  "content": "完整发言正文"
}
```

### 8.2 正方与反方输出增强

正方、反方阶段建议增加：

```json
{
  "strongest_claim": "本方最强论点",
  "supporting_reasons": ["理由 1", "理由 2"],
  "weaknesses": ["本方可能的薄弱点"]
}
```

文章辩论中可额外要求：

```json
{
  "evidence_assessment": "对文章证据或论证质量的判断"
}
```

话题辩论中不强制做文章证据审查。

### 8.3 主持人输出增强

主持人阶段建议增加：

```json
{
  "debate_focus": "本场辩论焦点",
  "disagreement_map": [
    {
      "issue": "争议点",
      "pro_position": "正方立场",
      "con_position": "反方立场"
    }
  ]
}
```

`disagreement_map` 是 V3 可视化的重要基础。

### 8.4 裁判输出增强

裁判阶段建议增加：

```json
{
  "verdict": "最终结论",
  "decision_basis": ["判定依据 1", "判定依据 2"],
  "credibility_score": 0.0,
  "winner": "pro | con | balanced"
}
```

裁判输出不需要包含继续追问问题。

### 8.5 兼容策略

为了兼容 V1/V2 历史数据：

- 后端读取旧消息时不要假设新字段必定存在。
- 前端展示结构化字段前先判断是否存在。
- Markdown 导出对缺失字段使用普通阶段正文兜底。
- 测试中保留旧格式兼容用例。

## 九、轻量配置能力

V3 增加用户可理解的轻量配置，不做复杂模型管理。

### 9.1 配置项

建议新增：

- 辩论深度：
  - 快速。
  - 标准。
  - 深度。
- 输出风格：
  - 简洁。
  - 详细。
- 话题辩论阶段：
  - 3 阶段极简。
  - 5 阶段标准。

文章辩论默认仍使用 9 阶段。

### 9.2 3 阶段话题辩论

3 阶段极简模式建议阶段：

1. 主持人定义辩题。
2. 正反观点合并陈述。
3. 裁判报告。

为了避免过早改动大量阶段枚举，可以在 V3 初版采用以下折中方案：

- 数据库仍复用已有阶段枚举。
- 3 阶段话题辩论映射到：
  - `moderator_opening`
  - `moderator_midpoint`
  - `judge_report`
- `moderator_midpoint` prompt 在极简模式下输出正反观点合并对照。

如果实现时发现该映射影响语义清晰度，再评估新增更明确的阶段枚举。

### 9.3 后端数据策略

轻量配置需要保存到辩论记录中，建议评估新增字段：

```text
debates.debate_depth
debates.output_style
debates.stage_mode
```

如果不希望 V3 引入迁移系统，也可以先将配置序列化到已有可扩展字段；但从长期维护看，建议在 V3 引入最小 Alembic 迁移或明确的 SQLite 轻量迁移脚本。

推荐取舍：

- 如果 V3 只面向本地开发演示，可以用手写 SQLite 迁移脚本。
- 如果后续还会继续迭代数据模型，建议正式引入 Alembic。

### 9.4 前端入口

配置入口建议放在创建表单中：

- 新建文章辩论页增加辩论深度、输出风格。
- 新建话题辩论页增加辩论深度、输出风格、阶段模式。
- 默认值保持当前 V2 行为：
  - 文章辩论：标准深度、详细输出、9 阶段。
  - 话题辩论：标准深度、简洁输出、5 阶段。

## 十、后端开发任务

### 10.1 报告导出支持

如果采用前端生成 Markdown，后端无需新增接口。

后端只需确保 `GET /api/debates/{debate_id}` 返回 Markdown 生成所需字段：

- 文章或话题信息。
- 辩论状态。
- 阶段消息。
- 裁判报告。
- winner。
- credibility_score。

### 10.2 整场重新运行 API

新增：

```text
POST /api/debates/{debate_id}/rerun
```

测试覆盖：

- 失败辩论可以重新运行。
- 已完成辩论可以重新运行，若产品决定支持。
- 重新运行会清理旧阶段消息。
- 重新运行会清理旧裁判结果。
- 文章辩论重新运行仍为 9 阶段。
- 话题辩论重新运行仍按原模式执行。

### 10.3 Agent schema 与 prompt 增强

任务：

- 扩展 Agent 输出 schema。
- 更新文章辩论 prompt。
- 更新话题辩论 prompt。
- 保留 JSON 修复重试。
- 增加新字段解析和兼容测试。

注意：

- 不要破坏旧消息读取。
- 不要让 prompt 输出过度冗长。
- 结构化字段应服务前端摘要、可视化和 Markdown 导出。

### 10.4 轻量配置后端支持

任务：

- 扩展创建辩论 schema。
- 扩展话题辩论创建 schema。
- 保存辩论深度、输出风格和阶段模式。
- 将配置注入 `article_payload` 或 orchestrator 上下文。
- 根据配置选择 prompt 或调整 prompt 输入。

测试覆盖：

- 默认配置兼容 V2 行为。
- 话题 3 阶段模式只执行 3 个阶段。
- 话题 5 阶段模式仍执行 5 个阶段。
- 文章辩论保持 9 阶段。

## 十二、数据库与迁移策略

如果新增字段，建议同步补充：

## 十三、验收标准

V3 完成时应满足：

- 首页能清楚解释产品用途、两类辩论模式和使用流程。
- 用户可以从首页快速进入文章辩论、话题辩论、文章库和最近辩论。
- 辩论详情页可以导出 Markdown。
- 辩论详情页可以复制 Markdown。
- Markdown 报告包含结论、可信度、正反方关键观点、关键分歧和阶段记录。
- Markdown 报告不包含继续追问环节。
- 失败辩论可以整场重新运行。
- 重新运行会清理旧阶段消息和旧裁判结果。
- 文章库支持普通文章 / 话题筛选。
- 文章库支持搜索和排序。
- 辩论详情页有更清晰的顶部摘要、阶段可视化和争议点展示。
- Agent 输出包含 `summary`、`key_points`、`content` 等基础结构化字段。
- 正反方输出包含最强论点、支持理由和薄弱点。
- 主持人输出可提供争议点矩阵。
- 裁判输出包含结论、判定依据、可信度和胜负结果。
- 创建辩论时可以选择辩论深度和输出风格。
- 话题辩论可以选择 3 阶段极简或 5 阶段标准。
- 默认配置兼容 V2 行为。
- 后端测试通过。
- 前端 lint/build 通过。

## 十四、建议实施顺序

1. 明确是否新增数据库字段，以及采用 Alembic 还是轻量迁移脚本。
2. 后端扩展辩论创建 schema，支持轻量配置。
3. 后端增强 Agent 输出 schema 和 prompt。
4. 后端新增整场重新运行接口。
5. 后端补齐配置、重跑、schema 兼容测试。
6. 前端重构首页，增加产品介绍和新手路径。
7. 前端升级创建表单，接入辩论深度、输出风格和话题阶段模式。
8. 前端升级详情页顶部摘要、阶段可视化和争议点展示。
9. 前端实现 Markdown 生成、复制和下载。
10. 前端升级文章库筛选、搜索和排序。
11. 运行后端 pytest。
12. 运行前端 lint 和 build。
13. 补充 V3 交接文档。

## 十五、开发交接检查清单

V3 交接时建议说明：

- 本次修改涉及的后端 API。
- 是否新增或修改数据库字段。
- 旧数据库如何升级或是否需要重建。
- 是否改变 LLM prompt 和输出 schema。
- 是否影响文章辩论 9 阶段流程。
- 是否影响话题辩论 5 阶段流程。
- 话题 3 阶段模式的阶段映射策略。
- Markdown 导出是在前端生成还是后端生成。
- 整场重新运行的清理规则。
- 已运行的后端测试结果。
- 已运行的前端 lint/build 结果。
- 已知限制和下一步建议。

## 十六、一句话总结

V3 的重点不是把系统扩成复杂平台，而是完成产品化闭环：用户知道怎么开始，过程看得懂，结果能导出，失败能重跑，Agent 输出更稳定、更适合可视化。
