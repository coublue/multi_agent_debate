# V5 开发计划：首页信息架构收敛与产品工作台增强

本文档用于明确 `multi_agent_debate` 第五阶段开发范围、产品意图、设计方向、实施阶段与验收标准。计划文件名确认为 `docs/v5-development-plan.md`。

V5 继承 V4 已完成的暗色 Notion AI 风格、多 Agent 文章/话题辩论、文章库、历史辩论、过程可视化、裁判报告和 Markdown 导出能力。本轮不重新设计整套视觉语言，重点是减少首页重复入口、明确主次关系，并把首页从“功能宣传页”推进为更实用的辩论工作台。

## 一、版本目标与产品意图

V5 的核心目标：

- 精简首页顶部导航，消除重复的创建入口。
- 保留文章库和历史辩论两个稳定的信息入口，使导航更像工作台导航而非功能目录。
- 继续沿用 V4 的黑底、高对比、紫色强调和 AI 编辑器式视觉，不另起一套风格。
- 优化首页首屏的操作层级，让“文章辩论”和“话题辩论”仍然容易开始，但不在顶栏重复出现。
- 增强回访用户的工作连续性，让用户更快找到进行中、最近完成或常用的辩论。
- 支持用户从一场已完成辩论的裁判结果继续提出新问题，并生成具有父子关系、可独立查看和运行的新辩论。
- 补充少量高价值产品能力，优先改善复用、整理和继续分析，而不是单纯增加展示卡片。

## 二、V5 设计简报

### 2.1 要解决的问题

首页当前同时承担品牌介绍、三个顶部操作入口、两个首屏创建入口、编辑器演示、功能卡片、统计概览和示例话题，信息密度偏高，并存在入口重复：

- 顶栏“开始辩论”与首屏“开始文章辩论”指向同一路由。
- 顶栏“话题辩论”与首屏“快速话题辩论”、示例话题卡片承担相似任务。
- 六张功能卡片中部分内容是能力说明，并不是需要长期占据首页的直接操作。
- 工作台数据有价值，但目前与宣传内容并列，回访用户的任务优先级不够突出。

### 2.2 视觉来源

- 继续使用 V4 已建立的暗色 Notion AI 风格和现有 Tailwind 体系。
- 继续使用黑色页面背景、深灰面板、紫色品牌强调、蓝色输入焦点状态。
- 优先复用现有卡片、按钮、输入框和状态样式，不引入大型 UI 组件库。
- V5 的设计重点是布局、层级、密度和交互反馈，不是更换色板或制作新的装饰体系。

### 2.3 交互级别

V5 按“完整交互”规划：保留真实路由、数据加载、失败提示、筛选、创建和详情跳转，不制作仅供展示的静态页面。

### 2.4 设计判断

- 顶部导航只保留品牌、文章库、历史辩论，形成稳定的三级结构：主页、内容资产、运行记录。
- 文章辩论和话题辩论继续作为首屏主操作，但通过主按钮与次按钮区分层级。
- 首页下半部分减少“功能说明卡片墙”，增加“正在进行”和“最近完成”等更贴近任务的内容。
- 示例话题保留为新用户启动辅助，但弱化为次级区域，避免与首屏创建入口竞争。
- 移动端不继续隐藏全部导航入口；应提供紧凑但可访问的文章库和历史辩论入口。

## 三、用户确认的需求

### 3.1 删除首页右上角按钮

- 删除首页顶栏右侧“开始辩论”按钮。
- 顶栏重新调整为品牌与文字导航的两端布局。
- 删除按钮后不能留下不自然的空白或移动端对齐问题。

### 3.2 精简首页顶部文字入口

- 顶部文字导航只保留“文章库”和“历史辩论”。
- 删除顶部“话题辩论”入口。
- 本项只移除首页顶部入口，不删除话题辩论功能、路由、表单、示例话题或首屏次操作。

## 四、建议纳入 V5 的前端设计更新

### 4.1 首页首屏层级收敛

- 保留产品标题与一句价值说明，但缩短正文，避免与右侧演示窗口争夺注意力。
- 保留“开始文章辩论”为主操作。
- 保留“快速话题辩论”为次操作，使用描边按钮或文字按钮。
- 将 `Alpha`、星号等纯装饰元素进一步克制化，优先保证界面像可用产品而不是概念落地页。
- 右侧编辑器预览继续作为产品机制说明，但其按钮和文案需要避免让用户误以为这是完整输入表单。

### 4.2 工作台优先于功能宣传

- 将“辩论概览”和“最近记录”前移或提高视觉权重。
- 将最近记录按“进行中”和“最近完成”区分，使回访用户能快速继续任务。
- 将后端英文状态值转换为统一中文状态标签，不直接显示 `pending`、`running`、`completed`、`failed`。
- 为统计加载失败增加明确的“重新加载”操作，而不是只显示错误文字。

### 4.3 功能卡片减量

- 将六张功能卡片压缩为三组核心价值：多 Agent 交锋、争议结构化、结论与报告。
- 删除与首屏操作重复的“文章深度辩论”“快速话题辩论”宣传卡片，或将其合并为一个“创建辩论”说明区。
- 减少 `AI`、`Q`、`VS`、`MAP` 等缩写标记的混用，优先使用一致的图标库或简洁文字标签。

### 4.4 响应式导航与可访问性

- 桌面端顶栏只显示文章库、历史辩论。
- 移动端提供同等可达的两个入口，不能因为 `md:flex` 隐藏后完全失去导航。
- 所有交互控件补齐清晰的 hover、focus-visible、disabled 和 loading 状态。
- 检查中英文混排、标题换行、按钮触控面积、文字对比度和键盘导航。

### 4.5 全站导航一致性

- 抽取或统一全站页面头部，减少首页、文章库、历史辩论、详情页分别维护导航造成的文案和样式漂移。
- 当前页面应有轻量激活态或上下文提示。
- 创建页之间仍可互相切换，但不把创建入口重新堆回全局顶栏。

## 五、建议补充的产品功能需求

以下内容按优先级分为“推荐纳入 V5”和“候选后续版本”，避免本轮范围失控。

### 5.1 推荐纳入 V5：继续工作区

首页新增“继续工作”区域：

- 优先展示进行中或等待中的辩论。
- 其次展示最近完成的辩论。
- 每条记录显示标题、类型、中文状态、更新时间和主要动作。
- 无记录时显示可执行空状态，分别引导文章辩论和话题辩论。

价值：让首页从一次性宣传页变成回访用户真正会使用的工作台。该能力可优先复用现有 `listDebates` 数据，不强制修改后端。

### 5.2 V5 核心目标：从结果继续追问

- 在已完成辩论详情页提供“继续追问”入口。
- 用户输入新的追问后，创建一场关联的独立子辩论，而不是修改、覆盖或向原辩论追加消息。
- 子辩论复用原文章或话题资源，默认继承父辩论的深度、输出风格和阶段模式。
- 子辩论以当前追问覆盖原关注问题，并只携带父辩论裁判报告的必要摘要、判定依据、关键分歧、可信部分和存疑部分。
- 新辩论继续复用现有 `BackgroundTasks -> debate_service.run_debate -> DebateOrchestrator` 执行链路，不建立第二套 Agent 编排系统。
- 新旧辩论应显示来源关系并支持跳转，允许形成多级追问链路。
- 父辩论运行结果保持不变；子辩论拥有独立状态、阶段消息、裁判报告、失败重跑和 Markdown 导出。

价值：当前流程在裁判报告处结束，但真实分析往往会产生下一轮问题。该能力能把一次性生成升级为连续推理工作流。

实现原则：继续追问是“带受控历史上下文的新辩论”，不是聊天式无限上下文。禁止把父辩论全部 Agent 消息或完整祖先记录直接拼接进 prompt，避免上下文和成本随追问次数持续膨胀。

### 5.3 候选后续版本：模板与预设管理

- 将首页硬编码示例话题升级为可维护的辩论模板。
- 模板包含标题、背景、关注问题、阶段模式、深度和输出风格。
- 支持从模板快速创建，也可保存用户自己的预设。

### 5.4 候选后续版本：报告对比

- 允许选择两场关联辩论，对比胜方、可信度、关键分歧和裁判依据。
- 优先用于同一文章不同问题、同一话题不同阶段模式的结果比较。

## 六、建议的 V5 范围边界

### 6.1 必做范围

- 首页右上角“开始辩论”按钮删除。
- 首页顶部“话题辩论”文字入口删除。
- 顶部仅保留文章库、历史辩论，并补足移动端可达性。
- 首页首屏与工作台信息层级优化。
- 最近记录中文状态化、错误重试和空状态优化。
- 功能卡片减量与文案统一。
- 全站导航样式与结构统一。
- 已完成辩论支持“继续追问”，创建关联但相互独立的子辩论。
- 子辩论保存当前追问、父辩论 ID 和创建时的父结果上下文快照。
- 继续追问复用现有 DeepSeek Agent 编排、阶段持久化、轮询、失败重跑和报告展示链路。

### 6.2 暂不实施

- 辩论收藏与置顶。
- 模板与预设管理。
- 报告对比。
- 登录、权限、多用户协作、云端同步。
- 修改 LLM 供应商或 Agent 底层模型配置。

用户已确认“辩论收藏与置顶”不纳入 V5；“继续工作区”和“从结果继续追问”均属于 V5 必做范围。

## 八、分区域实施方案

### 8.1 前端与设计

- 抽取共享头部并统一桌面、移动端导航。
- 重构首页区域顺序、操作层级和卡片数量。
- 将状态映射抽成可复用展示逻辑。
- 增加继续工作区、重试状态和可执行空状态。
- 在已完成辩论的裁判结果区域提供“继续追问”入口和问题输入界面；提交期间禁用按钮，避免重复创建。
- 创建成功后直接跳转到子辩论详情页，并沿用现有轮询展示运行进度。
- 在子辩论详情中显示“基于哪场辩论继续追问”，提供父辩论跳转；后续可扩展同一父辩论的子问题列表。
- 保持现有文章和话题创建表单的 API 行为。

### 8.2 后端

- 新增 `POST /api/debates/{debate_id}/follow-ups`，只允许基于 `completed` 状态创建追问。
- 新增 `DebateFollowUpCreate` 请求 schema，至少包含非空 `question`，并允许按需覆盖深度、输出风格和阶段模式。
- 新增 `create_follow_up_debate()` 服务函数：读取父辩论、复用 `article_id`、继承配置、构建上下文快照并创建 `pending` 子辩论。
- 创建完成后复用现有后台执行入口 `run_debate_background()`，不复制 `run_debate()` 或 `DebateOrchestrator`。
- 调整 `_article_payload()`：子辩论使用 `follow_up_question` 覆盖原 `Article.user_question`，同时注入 `follow_up_context`。
- Prompt 必须明确：历史裁判结论只是待检验背景，不得直接继承上一轮胜负判断，也不得简单重复上一轮报告。

### 8.3 数据

- `debates` 新增可空 `parent_debate_id`、`follow_up_question` 和 JSON 类型 `parent_context_snapshot`。
- 普通辩论三个字段均为空；子辩论保存直接父辩论 ID、当前问题和创建时的父结果裁剪快照。
- 上下文快照只保留 `main_claim`、`verdict`、`final_summary`、`decision_basis`、`key_disagreements`、`credible_parts`、`questionable_parts` 等必要字段，并对数组数量和文本长度设限。
- 子辩论复用父辩论的 `article_id`，不得修改或复制 `Article.user_question`，避免污染同一文章下的其他历史辩论。
- 删除父辩论时必须定义子辩论处理策略：V5 默认保留子辩论和上下文快照，将直接父关联置空或由服务层安全处理，禁止无提示级联删除整条分析链。
- SQLite 轻量迁移必须兼容旧数据库，新增字段允许为空，旧数据不受影响。
- 不在 V5 默认引入 Alembic；若本轮出现两个及以上数据结构变更，应重新评估正式迁移工具。

### 8.4 LLM 与 Agent

- 继续追问仍使用 DeepSeek OpenAI-compatible：`DEEPSEEK_API_KEY`、`https://api.deepseek.com`、`deepseek-v4-pro`。
- 若引入 LangChain，使用 `langchain_openai.ChatOpenAI`。
- 继续追问只传递必要摘要，避免直接拼接全部历史 Agent 输出导致上下文和成本膨胀。
- 现有 `DebateOrchestrator.run_iter()` 阶段顺序保持不变；主持人从当前问题重新定义辩题，正反方重新论证，裁判独立给出新报告。
- `follow_up_context` 通过现有 `article_payload` 进入 Agent 上下文，不新增第五个 Agent，也不增加独立模型调用链。

- 首页无明显重复卡片和竞争性主按钮。
- 键盘可以访问导航、按钮和列表项，focus-visible 清晰。
- 只有已完成辩论可以继续追问；运行中、等待中和失败状态返回明确错误。
- 空白追问不能提交，不存在的父辩论返回 `404`。
- 子辩论复用原 `article_id`，继承父辩论配置，同时不修改父辩论和原文章。
- 子辩论使用当前追问覆盖原关注问题，并只携带经过裁剪的父结果上下文。
- 新辩论不会覆盖旧记录，父子关联跳转正确，多级追问可形成稳定链路。
- 子辩论继续使用现有阶段持久化、详情轮询、失败重跑和 Markdown 导出能力。
- 快速连续点击不会重复创建子辩论；前端禁用提交，后端评估请求幂等保护。

## 十一、风险、假设与处理

- **范围膨胀**：继续工作区和继续追问已经纳入必做范围；收藏、模板和报告对比留在后续版本。
- **首页过度删减**：删除顶部入口不等于删除话题辩论能力；首屏和示例入口继续保留。
- **共享导航改造风险**：抽取组件时要检查所有页面的返回入口和当前页语义，避免机械替换。
- **移动端入口缺失**：当前首页导航在小于 `md` 时隐藏，V5 必须补充移动端可达方案。
- **SQLite 迁移风险**：新增字段要兼容已有本地数据库，不假设用户会删除数据库重建。
- **继续追问成本**：必须使用摘要和字段化上下文，不直接传递全量历史消息或完整祖先链。
- **历史结论锚定**：Prompt 必须声明父裁判结论只是待检验背景，新一轮主持人、正反方和裁判需要独立分析。
- **追问链删除关系**：父记录删除不能静默级联删除子辩论；上下文快照需要保证子辩论仍可解释和运行。
- **视觉参考边界**：继续借鉴 V4 已确认的风格，不直接复制外部产品代码或受限素材。

## 十二、用户检查点

1. 当前检查点：审阅并确认本计划、设计简报和 V5 功能边界。
2. 实现完成后：用户在本地检查桌面端和移动端页面。
3. 用户确认验收后：编写 `docs/v5-development-handoff.md`，更新必要文档并进行本地 Git 版本归档。
4. 只有用户明确要求时才推送 GitHub。
